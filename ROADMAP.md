# Wall Family Cookbook — Development Roadmap

> A design document for the next development steps. Written March 2026.

---

## 1. Repository Technical Summary

### What This Is

The Wall Family Cookbook is a private, password-protected recipe website built for the Wall family. It lives on Replit and is backed up to GitHub. Family members log in with a shared passphrase and can browse, add, edit, and delete recipes — all through their browser.

### How It Works

```
Browser
  │
  ├── GET /           → Express serves a dynamically assembled HTML page
  │                     (template + recipe cards injected from the database)
  │
  ├── POST /api/login → Cookie-based auth (30-day session)
  │
  ├── POST /api/add-recipe  ─┐
  ├── POST /api/edit-recipe  ├── Calls Google Gemini 2.5 Flash → generates
  ├── POST /api/chat         ┘   recipe card HTML → stores in PostgreSQL
  │
  └── POST /api/delete-recipe / save-card-html / get-card-html
```

### Key Files

| File | Purpose |
|---|---|
| `server.js` | Express backend (~900 lines). Handles auth, all API routes, AI calls, DB queries, and HTML page assembly. |
| `index.html` | The entire frontend (~1500 lines). All CSS, HTML structure, and JS in one file. No build step. |
| `package.json` | Declares `express`, `@google/generative-ai`, and `pg`. Start command: `npm start`. |
| `replit.md` | Internal Replit project documentation. |
| `.replit` | Replit run/deployment config (gitignored; lives only on Replit). |

### Technology Stack

- **Runtime**: Node.js 20
- **Backend**: Express.js (no framework, just routing + middleware)
- **Frontend**: Vanilla HTML/CSS/JavaScript — zero dependencies, no build step
- **Database**: PostgreSQL 16 (Replit managed, connection via `DATABASE_URL` env var)
- **AI**: Google Gemini 2.5 Flash (via `GEMINI_API_KEY` env var)
- **Hosting**: Replit (autoscale deployment, port 5000 → 80)

### Data Model (Current)

Recipes are stored as AI-generated HTML blobs in a PostgreSQL table:

```sql
CREATE TABLE recipes (
  id          SERIAL PRIMARY KEY,
  category    TEXT NOT NULL,
  author_name TEXT,
  card_html   TEXT NOT NULL,       -- Full flip-card HTML generated by Gemini
  card_id     TEXT UNIQUE NOT NULL, -- e.g. "card-grandmas-apple-pie"
  created_at  TIMESTAMPTZ DEFAULT now()
);
```

### Authentication

A single shared passphrase (family password) gates the entire site. On login, the server sets an HMAC-signed HttpOnly cookie that expires after 30 days.

#### 3. Fix mobile card grid overflow
**Why it matters**: The recipe card grid uses `minmax(320px, 1fr)`. On phones narrower than ~384px (accounting for padding), cards overflow the screen horizontally. Changing to `minmax(min(320px, 100%), 1fr)` fixes this.

**Why it matters**: As the cookbook grows, scrolling through all categories to find one recipe becomes tedious. A simple client-side text filter (no server changes needed) would let family members type a recipe name and instantly see matches.
**Effort**: 2–3 hours
**Implementation hint**: Add an `<input>` to the nav bar. On `input` event, loop over `.flip-card` elements and toggle `display: none` based on whether the card's `.front-title` text contains the search string.

#### 5. Recipe JSON export / backup
**Why it matters**: All recipes live in Replit's PostgreSQL database. If the Replit project is deleted or the database is reset, recipes are gone. A one-click "Download recipes.json" button in the UI (admin only, or always available) would let the family keep a backup.
**Effort**: 2 hours
**Implementation hint**: Add `GET /api/export` that returns all rows as JSON. Add a small "Export" button in the nav or footer that triggers a download.

### P2 — Nice to Have

#### 6. Persist author name in browser localStorage
**Why it matters**: Every time a family member adds a recipe, they must re-type their name in the "Your Name" field. Saving it to `localStorage` after first use would save a small but repeated annoyance.
**Effort**: 30 minutes
**Implementation hint**: On form submit, `localStorage.setItem('wfc_author', authorName)`. On modal open, pre-fill the field with `localStorage.getItem('wfc_author')`.

#### 7. Accessible keyboard navigation
**Why it matters**: Flip cards currently flip only on click. Keyboard users (and some mobile assistants) cannot flip them. Adding `tabindex="0"` and a `keydown` handler (Enter/Space to flip) makes the site usable without a mouse.
**Effort**: 1 hour

#### 8. Structured recipe data alongside HTML
**Why it matters**: Recipes are stored as raw HTML blobs. This makes it impossible to query or transform them (e.g., for export, search, or a future mobile app). Adding a structured JSON field alongside the HTML would give future flexibility without a breaking migration.
**Effort**: 4–6 hours
**Implementation hint**: Add a `recipe_json` JSONB column to the `recipes` table. Have Gemini also return structured JSON alongside the card HTML. See Section 5 for the proposed schema.

### Issue 3: Fix mobile card grid horizontal overflow

**Title**: Bug: Recipe cards overflow screen on phones narrower than 384px

**Problem**
The card grid uses `minmax(320px, 1fr)`. With the section's 2rem left+right padding, the usable width on a 375px-wide phone is ~311px — less than the 320px minimum. This causes a horizontal scrollbar and clipped cards.

**Proposed Solution**
Change `index.html` line 254:
```css
/* Before */
grid-template-columns: repeat(auto-fill, minmax(320px, 1fr));

/* After */
grid-template-columns: repeat(auto-fill, minmax(min(320px, 100%), 1fr));
```
`min(320px, 100%)` means "use 320px, but never more than the available container width."

**Acceptance Criteria**
- [ ] On a 375px-wide viewport, cards fill the full width without horizontal scroll
- [ ] On desktop (≥ 768px), layout is unchanged
- [ ] No new visual regressions on tablet

**Effort**: 15 minutes
**Labels**: `bug`, `mobile`, `good first issue`

## 6. Recipe Editing Workflow

### Current Workflow (Option A — Already Implemented)

The site already implements the optimal workflow for a small family:

1. Family member clicks **"Add a Recipe"** in the nav bar
2. Fills in: category, their name, and either:
   - **Paste recipe text** (from a card, magazine, memory)
   - **Paste a URL** (from AllRecipes, NYT Cooking, a food blog, etc.)
3. Clicks **"Add to Cookbook"**
4. The server sends the content to Gemini 2.5 Flash, which generates a beautifully formatted flip card
5. The page reloads and the new card appears in the correct section

This is already the simplest possible experience. No account needed, no formatting required, and the AI handles every presentation detail.

### Option B — Approval Queue (Not Recommended)

An approval queue would require:
- A separate "pending recipes" table
- An admin dashboard
- Email or notification to the repo owner
- Extra review step before a recipe appears

For a private family site of ~10 people, this adds friction with no meaningful benefit. Everyone who has the passphrase is trusted.

### Recommendation: Keep Option A, Add Two Small UX Improvements

**Improvement A — Persist author name in localStorage** (30 min)

Each family member only needs to type their name once. See Issue 6 for implementation details.

```js
// In openAddModal():
const saved = localStorage.getItem('wfc_author');
if (saved) document.getElementById('recipe-author').value = saved;

// In submitRecipe(), on success:
localStorage.setItem('wfc_author', form.authorName.value.trim());
```

**Improvement B — Show character/URL hints in the textarea** (15 min)

The current placeholder text says "Paste the full recipe here, or paste a URL…". Adding a note about URL support directly in the modal label (not just the placeholder, which disappears on focus) would help non-technical family members discover the URL import feature.

### Acceptance Criteria for Current Workflow

- [ ] Family member can add a recipe in under 2 minutes
- [ ] Recipe appears correctly categorized after page reload
- [ ] Author name is attributed correctly on the card
- [ ] Error message is shown if the AI call fails (e.g., API key invalid)
- [ ] Long AI calls (up to ~15 seconds) don't show a blank/frozen screen

The project runs on Replit with:
- **Run command**: `node server.js` (also `npm start`)
- **Port**: 5000 (internal) mapped to 80 (external)

The `.gitignore` file excludes `.replit` and `replit.nix`. This is intentional: Replit auto-generates and manages these files. Committing them can cause conflicts when Replit updates its own config. The project runs fine without them in git because Replit handles the configuration on its side.

[deployment]

## 10. Security and Privacy Notes

### Risk 1: Passphrase Exposed in Source Code

**Risk**: The passphrase `"Joe+Linda"` was a string literal in `server.js`. Anyone with read access to the repo could see it.
**Mitigation**: Use `process.env.PASSPHRASE` with a fallback. Store the real value in Replit Secrets. Never commit the actual passphrase to Git. ✅ Fixed in this PR.

### Risk 2: Gemini API Key Exposed

**Risk**: If `GEMINI_API_KEY` is ever hardcoded or accidentally committed, attackers could use the family's API quota to run unlimited AI requests.
**Mitigation**: Always store the key in Replit Secrets (or `.env` locally, which is gitignored). Periodically rotate the key in Google AI Studio. Set spending limits on the Google Cloud project.

### Risk 3: HTML Injection via AI-Generated Content

**Risk**: Recipes are stored as raw HTML. If the AI ever generates malicious `<script>` tags (whether by accident or through prompt injection in a pasted recipe), those could execute in other users' browsers.
**Mitigation**: `server.js` already strips `<script>` tags and forbids `javascript:` URLs from AI output. The existing `sanitizeHtml()` function covers the main attack vectors. Continue keeping this in place for all HTML paths (AI-generated and direct-edit).

### Risk 4: Direct HTML Edit Mode

**Risk**: The "Edit HTML directly" tab in the edit modal allows saving arbitrary HTML to the database. A malicious family member (or someone who guesses the passphrase) could inject content.
**Mitigation**: The existing `sanitizeHtml()` function in `server.js` runs on all saved HTML, including direct edits. It blocks `<script>`, `<iframe>`, `<object>`, `<embed>`, `<link>`, `<meta>`, and `javascript:` attributes. This is sufficient for a private family site. If the site ever becomes more public, consider stricter allow-list-based sanitization (e.g., via the `sanitize-html` npm package).

### Risk 5: Database Connection String Exposure

**Risk**: `DATABASE_URL` contains credentials. If it appears in logs, error messages, or is committed to Git, the database is exposed.
**Mitigation**: Never log `DATABASE_URL`. Ensure `.env` is in `.gitignore` (it is). On Replit, `DATABASE_URL` is auto-injected as a secret and never appears in the repo. The existing code reads it via `process.env.DATABASE_URL` correctly.
